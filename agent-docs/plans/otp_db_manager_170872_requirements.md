# OTP Database Connection Manager — Requirements (v3)

## Goal

Build a robust, self-healing OTP supervisor tree that manages SQLite connections for both the registry DB and per-family DBs. The design prioritises:
- Composability and flat, readable Gleam code (`use`, `result` pipelines, no deep nesting)
- SQL isolated to `.sql` files — all queries go through parrot-generated Gleam modules
- Named process discovery (no Subject passing between modules)
- True OTP restart semantics via `gleam/otp/static_supervisor`
- A generic typed query API for the family DB actor

---

## Parrot Integration

### How parrot works (from source inspection)

Parrot reads `.sql` files and generates a Gleam module with:
- A **record type** per `:one`/`:many` query (named after the query in PascalCase)
- A **function** per query that returns `#(sql_string, [Param], Decoder)` (or `#(sql_string, [Param], "")` for `:exec`)
- A **decoder function** per `:one`/`:many` query

The caller pattern (as seen in the existing `backend.gleam`):
```gleam
let #(sql, params, decoder) = sql.get_account_by_email(email)
let params = list.map(params, parrot_to_sqlight)
sqlight.query(sql, on: conn, with: params, expecting: decoder)
```

### Two parrot-generated modules

We need two generated modules — one per database:

| SQL files | Generated module | Used by |
|---|---|---|
| `src/sql/registry_queries.sql` + `registry_schema.sql` | `src/backend/registry_sql.gleam` | `registry_actor.gleam` |
| `src/sql/queries.sql` + `schema.sql` | `src/backend/sql.gleam` (already exists, stale) | `family_db_actor.gleam` |

### Running parrot

Parrot needs a real SQLite file to introspect the schema. Two invocations are needed:

```sh
# Registry queries — needs registry.db to exist with schema applied
gleam run -m parrot -- --sqlite data/registry.db

# Family queries — needs a family DB to exist with schema applied
gleam run -m parrot -- --sqlite data/family_template.db
```

Parrot reads its output path from a config file (`.parrot.json` or `parrot.json` at project root). We need to understand if it supports multiple output configs or if we need two separate invocations with different configs.

**Action for Writer:** Before writing code, run parrot against both DBs and confirm what gets generated. The generated modules are the ground truth for how actors call queries.

### `parrot_to_sqlight` helper

The existing `parrot_to_sqlight` function in `backend.gleam` maps `parrot/dev.Param` → `sqlight.Value`. This helper needs to move to a shared module (e.g. `backend/db.gleam`) so both `registry_actor.gleam` and `family_db_actor.gleam` can use it.

## Architecture

### Named Process Discovery

Both core actors register themselves under well-known names. Rather than Erlang atom registration (which requires Pid → Subject reconstruction), we use a simpler pattern: **each actor stores its own Subject in a module-level process dictionary or ETS table on start**.

**Simplest approach for this project:** Pass `process.Name` values created once in `main()` through the supervisor worker closures. The `actor.named(name)` API in gleam_otp registers the actor under that name and allows `process.named_subject(name)` lookups.

### Supervisor Tree

```
main()
  └─ static_supervisor (OneForOne)
        ├─ RegistryActor   — opens registry.db, runs DDL, registers as name
        └─ FamilyDbSupervisor — plain actor, registers as name

DbEvictor — spawned as linked process after supervisor starts
```

## Style Requirements (non-negotiable)

1. **No deep nesting** — use `use` + `result.try` / `result.map` / `result.map_error` pipelines to flatten case chains
2. **Extract helpers** — if a block of logic repeats or nests > 2 levels, extract it into a named private function
3. **SQL stays in `.sql` files** — no raw SQL strings in `.gleam` files; all queries go through parrot-generated functions
4. **No `sqlight.exec` for parameterised queries** — use the parrot-generated functions + `db.exec_command` / `db.exec_query` helpers which call `sqlight.query` with the right decoder

## `backend/src/backend/db.gleam` — Shared DB helpers

This module provides:

```gleam
import gleam/list
import gleam/result
import parrot/dev
import sqlight

/// Convert a parrot Param to a sqlight Value.
/// Extracted from the old backend.gleam parrot_to_sqlight helper.
pub fn to_sqlight(param: dev.Param) -> sqlight.Value { ... }

/// Run a parrot-generated :one or :many query.
/// Takes the #(sql, params, decoder) tuple from parrot and executes it.
pub fn exec_query(
  conn: sqlight.Connection,
  query: #(String, List(dev.Param), decode.Decoder(a)),
) -> Result(List(a), String) {
  let #(sql, params, decoder) = query
  sqlight.query(sql, on: conn, with: list.map(params, to_sqlight), expecting: decoder)
  |> result.map_error(fn(e) { e.message })
}

/// Run a parrot-generated :exec query (no rows returned).
pub fn exec_command(
  conn: sqlight.Connection,
  query: #(String, List(dev.Param), String),
) -> Result(Nil, String) {
  let #(sql, params, _) = query
  sqlight.query(sql, on: conn, with: list.map(params, to_sqlight), expecting: decode.success(Nil))
  |> result.map(fn(_) { Nil })
  |> result.map_error(fn(e) { e.message })
}
```

## `registry_actor.gleam` — Message protocol

```gleam
pub type Message {
  GetAccountByEmail(email: String, reply_to: Subject(Result(RegistrySql.GetAccountByEmail, Nil)))
  GetDbPathForFamily(family_id: String, reply_to: Subject(Result(String, Nil)))
  InsertRegistryAuthToken(account_id: Int, token_hash: String, token_type: String, expires_at: String, used_at: Option(String), created_at: String, reply_to: Subject(Result(Nil, String)))
  GetRegistryAuthTokenByHash(hash: String, reply_to: Subject(Result(RegistrySql.GetRegistryAuthTokenByHash, Nil)))
  MarkRegistryAuthTokenUsed(id: Int, used_at: String, reply_to: Subject(Result(Nil, String)))
  DeleteExpiredRegistryAuthTokens(before: String, reply_to: Subject(Result(Nil, String)))
}
```

Note: Return types use the parrot-generated record types (e.g. `registry_sql.GetAccountByEmail`) rather than manually defining equivalent types. This ensures the types stay in sync with the SQL.

## `family_db_actor.gleam` — Message protocol

```gleam
pub type Message {
  Exec(sql_query: #(String, List(dev.Param), String), reply_to: Subject(Result(Nil, String)))
  GetLastUsedAt(reply_to: Subject(Int))
  Shutdown
}
```

The `Exec` message takes the raw parrot 3-tuple (for `:exec` queries). Route handlers will call parrot functions then send the result to the actor. A typed `Query` variant will be added when the first routes are built.

## Init sequence — no deep nesting

Actor initialisers use `use` + `result.try` throughout:

```gleam
// registry_actor init:
use conn <- result.try(open_db(db_path))
use _ <- result.try(enable_foreign_keys(conn))
use _ <- result.try(run_ddl(conn))
actor.initialised(State(conn:)) |> actor.returning(subject) |> Ok

// family_db_actor init:
use db_path <- result.try(fetch_db_path(family_id, registry_name))
use conn <- result.try(open_db(db_path))
use _ <- result.try(enable_foreign_keys(conn))
use _ <- result.try(run_family_ddl(conn))
actor.initialised(State(family_id:, db_path:, conn:, last_used_at: now_ms())) |> actor.returning(subject) |> Ok
```

Each helper (`open_db`, `enable_foreign_keys`, `run_ddl`) is a small private function returning `Result(_, String)`.

## Family schema DDL

The 14-table DDL from `schema.sql` is read and run using `sqlight.exec` (which handles multiple statements). This is the **only** place where raw SQL is used directly — because DDL is schema management, not a query, and parrot doesn't generate DDL runners. The DDL content itself lives in `schema.sql`; the actor reads it as a constant (embedded at compile time via a Gleam `const`).
