# Registry DB Startup Initialization — Requirements

## Goal

On backend startup, initialize the central registry SQLite database before the HTTP server begins accepting requests.

## Behavior

1. Read required environment variables — crash with a clear error message if any are missing:
   - `REGISTRY_DB_PATH` — path to the registry SQLite file (e.g. `data/registry.db`)
   - `PORT` — HTTP port to listen on (e.g. `3000`)

2. Open a SQLite connection to the path specified by `REGISTRY_DB_PATH`. Create the file if it does not exist (SQLite does this automatically).

3. Run `PRAGMA foreign_keys = ON` on the registry connection.

4. Execute the registry schema DDL (all `CREATE TABLE IF NOT EXISTS` statements from `registry_schema.sql`) to initialize tables if they don't exist. This is safe to run on every startup — idempotent.

5. If any step fails, crash hard with `let assert Ok(_)` — the registry is required for the app to function.

6. Start the HTTP server on the port from `PORT`.

## Environment Variables

| Variable | Required | Description |
|---|---|---|
| `REGISTRY_DB_PATH` | Yes | Path to registry SQLite file. Crash with clear message if missing. |
| `PORT` | Yes | HTTP port. Crash with clear message if missing. Must be parseable as an integer. |

## Library

- Use `envoy` (already in `manifest.toml` as a transitive dep) for reading env vars. Add it as a direct dependency in `gleam.toml`.
- Use `sqlight` (already a direct dep) for the DB connection.

## Code structure

- Add a `backend/config.gleam` module that reads all env vars and returns a typed `Config` record. Crash with a descriptive `panic` message if a required var is missing or invalid.
- Add a `backend/registry.gleam` module responsible for opening the registry DB connection and running the schema DDL on startup.
- Update `backend.gleam` (`main`) to: load config → initialize registry → start HTTP server on `config.port`.
- Remove the old hardcoded `run_startup_query` (the `list_greetings` call) from `backend.gleam` — it's stale now that the schema has changed.

## Registry Schema DDL (to embed as a string in registry.gleam)

The DDL to run on startup (copied from `registry_schema.sql`):

```sql
CREATE TABLE IF NOT EXISTS families (
  id         TEXT PRIMARY KEY,
  name       TEXT NOT NULL,
  db_path    TEXT NOT NULL UNIQUE,
  status     TEXT NOT NULL DEFAULT 'active',
  created_at TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS accounts (
  id            INTEGER PRIMARY KEY,
  family_id     TEXT NOT NULL REFERENCES families(id) ON DELETE CASCADE,
  email         TEXT NOT NULL UNIQUE,
  role          TEXT NOT NULL DEFAULT 'member',
  created_at    TEXT NOT NULL,
  last_login_at TEXT
);

CREATE TABLE IF NOT EXISTS registry_auth_tokens (
  id         INTEGER PRIMARY KEY,
  account_id INTEGER NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,
  token_hash TEXT NOT NULL,
  token_type TEXT NOT NULL,
  expires_at TEXT NOT NULL,
  used_at    TEXT,
  created_at TEXT NOT NULL
);
```

## What NOT to do

- Do not touch `backend/src/backend/sql.gleam` — it is auto-generated by parrot.
- Do not add a persistent registry DB connection to the HTTP request handler yet — that is a future task (OTP actor-based connection management).
- The registry connection opened at startup is used only for the DDL run, then closed. Persistent connection management is out of scope for this task.
